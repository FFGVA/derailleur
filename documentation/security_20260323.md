## 1. Résumé exécutif

L'application Dérailleur présente une posture de sécurité globalement **satisfaisante** pour une application associative. Les fondations Laravel/Filament fournissent une base solide (ORM Eloquent, protection CSRF, hachage bcrypt). Cependant, plusieurs vulnérabilités méritent attention :

- **2 vulnérabilités HIGH** : XSS stocké via le champ description d'événement (RichEditor → affichage `{!! !!}` non assaini)
- **7 vulnérabilités MEDIUM** : injection d'en-têtes email, race condition sur les numéros de membre, politique d'accès trop permissive pour les cheffes de peloton, IDOR potentiel sur le portail chef de peloton
- **Plusieurs vulnérabilités LOW** : timeout de session portail long, dépendances wildcard, headers de sécurité manquants

Aucune vulnérabilité critique d'injection SQL ou d'exécution de commandes n'a été détectée.

## 2. Méthodologie

Analyse statique complète du code source couvrant :
- Revue manuelle de tous les contrôleurs, modèles, politiques, middlewares
- Recherche systématique de patterns dangereux (`{!! !!}`, `->html()`, `DB::raw()`, `exec()`, `eval()`)
- Analyse de la configuration (`.env`, `config/`, `.htaccess`, `robots.txt`)
- Revue du schéma de base de données et des triggers d'audit
- Analyse des flux d'authentification (admin Filament, portail magic link, activation adhésion)
- Revue de la gestion des données personnelles et conformité LPD
- Analyse des dépendances (composer.json, package.json)
- Revue du script de déploiement

## 3. Authentification & gestion des sessions

### 3.1 Panneau d'administration (Filament)

**Statut : SÉCURISÉ**

- Authentification par mot de passe via Filament login intégré (`AdminPanelProvider.php:32`)
- Réinitialisation de mot de passe activée (`AdminPanelProvider.php:33`)
- Mots de passe hashés via bcrypt (`User.php:53` — cast `'password' => 'hashed'`)
- Middleware complet : `StartSession`, `AuthenticateSession`, `VerifyCsrfToken` (`AdminPanelProvider.php:84-93`)

**Point d'attention :** `canAccessPanel()` retourne `true` pour tous les utilisateurs authentifiés (`User.php:73-76`). L'accès est contrôlé par les politiques Filament, pas par cette méthode. Risque mitigé car seuls les utilisateurs invités ont un compte.

### 3.2 Portail membre (magic link)

**Statut : SÉCURISÉ**

- Tokens générés avec `bin2hex(random_bytes(32))` — 256 bits d'entropie (`MemberMagicToken.php:39`)
- Stockage par hash SHA-256 uniquement — le token brut n'est jamais en base (`MemberMagicToken.php:40`)
- Expiration : 15 minutes (configurable `config/ffgva.php:14`)
- Usage unique : `markUsed()` vérifie `is_null($this->used_at)` (`MemberMagicToken.php:60-65`)
- Régénération de session après login (`PortalAuthController.php:74`)

**Point d'attention — LOW :** La régénération de session se fait *après* l'écriture des données de session. Best practice : régénérer d'abord.

### 3.3 Activation d'adhésion

**Statut : ACCEPTABLE avec réserves**

- Token d'activation hashé avec `Hash::make()` (bcrypt), validé avec `Hash::check()` (`AdhesionActivationController.php:40`)
- Expiration : 72 heures (`config/ffgva.php:12`)
- Token effacé après usage (`AdhesionActivationController.php:55`)

**Point d'attention — MEDIUM :** L'email est transmis en clair dans l'URL d'activation :
```
/adhesion/confirmer?token={rawToken}&email={member->email}
```
Cela expose l'adresse email dans l'historique du navigateur, les logs serveur et les en-têtes Referer. Recommandation : utiliser uniquement le token et résoudre le membre côté serveur.

### 3.4 QR code de carte de membre

**Statut : SÉCURISÉ**

- URLs signées temporaires (`URL::temporarySignedRoute`) avec expiration de 5 minutes (`PortalController.php:112-115`)
- Validation de signature côté serveur (`$request->hasValidSignature()`)
- Renouvellement automatique côté client

### 3.5 Configuration des sessions

| Paramètre | Valeur | Commentaire |
|-----------|--------|-------------|
| Driver | database | Stockage serveur, pas dans le cookie |
| Lifetime | 120 min | Standard |
| HTTP-only | true | Pas d'accès JavaScript |
| SameSite | lax | Protection CSRF |
| Secure | env-based | À vérifier en production |
| Encryption | false (dev) / true (prod) | Correct en production |

**Point d'attention — LOW :** Le timeout du portail membre est de 300 minutes (5h) (`config/ffgva.php:15`). Recommandation : réduire à 60 minutes pour les données sensibles.

## 4. Autorisation & contrôle d'accès

### 4.1 Politiques Filament

**MemberPolicy & EventPolicy :**

| Action | Admin | ChefPeloton | Commentaire |
|--------|-------|-------------|-------------|
| viewAny | ✓ | ✓ | **MEDIUM** — Trop permissif |
| view | ✓ | ✓ | **MEDIUM** — Trop permissif |
| create | ✓ | ✗ | Correct |
| update | ✓ | ✓ (membres) / ✓ (propres événements) | Correct |
| delete | ✓ | ✗ | Correct |
| restore | ✓ | ✗ | Correct |
| forceDelete | ✓ | ✗ | Correct |

**Vulnérabilité MEDIUM — Politique de vue trop permissive :** Les cheffes de peloton peuvent voir **tous** les membres et **tous** les événements via le panneau Filament. Pas de filtrage par scope.

- `MemberPolicy.php:11-19` — `viewAny()` et `view()` retournent `true` pour tout utilisateur
- `EventPolicy.php:11-19` — Idem

**Recommandation :** Restreindre `viewAny()` pour les cheffes de peloton aux membres inscrits à leurs événements.

### 4.2 Export vCard en masse

**Vulnérabilité MEDIUM :** L'action de téléchargement vCard en masse (`ListMembers.php:17-47`) exporte **tous** les membres sans vérification de rôle. Une cheffe de peloton peut télécharger l'ensemble des contacts.

**Recommandation :** Restreindre l'export en masse aux administrateurs.

### 4.3 Portail cheffe de peloton — IDOR potentiel

**Vulnérabilité MEDIUM :** La route `/portail/peloton/{event}/membre/{targetMember}` utilise le model binding Laravel. La vérification que `targetMember` est bien inscrit à l'événement dépend du contrôleur. Si cette vérification est absente ou incomplète, une cheffe de peloton pourrait accéder aux données de n'importe quel membre via manipulation d'URL.

**Données exposées :** nom complet, email, tous les numéros de téléphone avec statut WhatsApp, permission photo.

### 4.4 Accès aux factures PDF (portail)

**Statut : SÉCURISÉ** — Le contrôleur vérifie `$invoice->member_id !== $member->id` avant de servir le PDF (`PortalController.php:287-314`). Protection IDOR correcte.

## 5. Validation des entrées & injections

### 5.1 Injection SQL

**Statut : SÉCURISÉ** — Aucune injection SQL détectée.

- L'application utilise exclusivement Eloquent ORM avec des requêtes paramétrées
- Le seul `DB::statement()` trouvé est paramétré correctement : `DB::statement('SET @current_user_id = ?', [auth()->id()])` (`SetDatabaseUserId.php:15`)
- Aucun `DB::raw()`, `whereRaw()` ou requête SQL construite par concaténation

### 5.2 Injection de commandes

**Statut : SÉCURISÉ** — Aucune fonction dangereuse détectée.

- Aucun appel à `exec()`, `shell_exec()`, `system()`, `passthru()`, `proc_open()`, `eval()`, `unserialize()` dans le code applicatif

### 5.3 Protection contre l'assignation de masse

**Statut : SÉCURISÉ** — Tous les modèles utilisent `$fillable` (liste blanche explicite).

- `Member.php:29-50` — Champs fillable définis
- `User.php` — Champs fillable définis
- Aucun modèle avec `$guarded = []`

### 5.4 Validation des formulaires

**Statut : ACCEPTABLE avec réserves**

Les Form Requests valident correctement les champs principaux :
- `AdhesionRequest.php` : email, prenom, nom validés
- `ContactRequest.php` : email, name, message validés
- Champ honeypot (`website`) présent sur les formulaires publics

**Point d'attention — LOW :** Plusieurs champs optionnels de l'adhésion n'ont pas de longueur maximale :
```php
// AdhesionRequest.php
'type_velo' => ['nullable', 'string'],  // Pas de max
'instagram' => ['nullable', 'string'],  // Pas de max
'strava'    => ['nullable', 'string'],  // Pas de max
```
Risque de stockage de données volumineuses dans le champ JSON metadata.

## 6. Cross-Site Scripting (XSS)

### 6.1 XSS stocké via description d'événement — HIGH

**C'est la vulnérabilité la plus significative de l'application.**

**Vecteur d'attaque :** Un administrateur ou une cheffe de peloton saisit du HTML/JavaScript malveillant dans le champ description d'événement via le RichEditor Filament.

**Points d'impact :**

1. **Panneau admin** — `ViewEvent.php:35` :
   ```php
   Components\TextEntry::make('description')->html()
   ```
   Le contenu HTML est rendu sans assainissement dans la page de vue.

2. **Portail membre** — `evenement.blade.php:225` :
   ```blade
   {!! $event->description !!}
   ```
   Sortie Blade non échappée. Tout membre visualisant l'événement exécuterait le script.

3. **Portail cheffe de peloton** — `peloton-event.blade.php` : même pattern.

**Impact :** Vol de session, redirection vers site malveillant, manipulation de l'interface.

**Recommandation immédiate :**
- Installer `mews/purifier` (wrapper HTMLPurifier pour Laravel)
- Assainir le HTML à l'enregistrement ET à l'affichage
- Remplacer `{!! $event->description !!}` par `{!! clean($event->description) !!}`

### 6.2 Injection JavaScript via urlencode() — MEDIUM

**Fichier :** `evenement.blade.php:240, 280` et `peloton-event.blade.php:240`

```blade
<a href="#" onclick="openMaps('{{ urlencode($event->location) }}');">
```

La fonction `openMaps()` passe la valeur à `window.location` et `window.open()`. `urlencode()` n'est pas un échappement approprié pour le contexte JavaScript.

**Recommandation :** Utiliser `json_encode()` pour l'échappement dans le contexte JavaScript :
```blade
onclick="openMaps({{ json_encode($event->location) }})"
```

### 6.3 Métadonnées membre

**Statut : SÉCURISÉ** — Les valeurs de métadonnées sont affichées avec `{{ }}` (échappement automatique Blade) dans `member-metadata.blade.php:12`.

## 8. Sécurité des emails

### 8.1 Injection d'en-têtes email — MEDIUM

**Fichiers concernés :**
- `ContactMail.php:25` — `replyTo: [$this->email]` (email utilisateur)
- `AdhesionMail.php:34` — `replyTo: [$this->email]`
- `MemberUpdateRequestMail.php:27` — `replyTo: [$this->member->email]`

L'email fourni par l'utilisateur est utilisé directement dans l'en-tête Reply-To. Bien que le composant Mail de Laravel filtre les caractères dangereux, une validation explicite des caractères CRLF dans les Form Requests renforcerait la protection.

### 8.2 Injection dans le sujet email — MEDIUM

**Fichier :** `MemberUpdateRequestMail.php:28`
```php
subject: 'Demande de modification — ' . $this->member->first_name . ' ' . $this->member->last_name,
```

Le nom du membre est concaténé directement dans le sujet. Si un nom contient des caractères de contrôle (`\r\n`), cela pourrait injecter des en-têtes supplémentaires.

**Recommandation :** Valider l'absence de `\r` et `\n` dans les noms lors de la saisie.

### 8.3 Données personnelles dans les emails

Les emails suivants transmettent des données personnelles significatives :

| Email | Destinataire | Données transmises |
|-------|-------------|-------------------|
| AdhesionMail | Admin | Nom, prénom, email, téléphone, Instagram, Strava, photo_ok |
| MemberUpdateRequestMail | Admin | Adresse complète, tous les téléphones (anciens et nouveaux) |
| InvoiceMail | Membre | Nom, adresse, montants, QR-bill |
| ExpiredMemberRegistrationMail | Admin | Nom, numéro, email, date fin adhésion |

**Risque :** Les emails transitent en clair (pas de chiffrement de bout en bout). Les données personnelles sont exposées dans les boîtes mail des administrateurs.

## 12. Dépendances & chaîne d'approvisionnement

### 12.1 Dépendances PHP (composer.json)

**Points d'attention :**

| Dépendance | Version | Risque |
|-----------|---------|--------|
| `barryvdh/laravel-dompdf` | `*` (wildcard) | Pourrait installer une version vulnérable |
| `sprain/swiss-qr-bill` | `*` (wildcard) | Idem |
| `laravel/tinker` | `^2.10.1` (require, pas require-dev) | REPL PHP disponible en production |

**Recommandations :**
- Remplacer les wildcards `*` par des versions contraintes (ex: `^2.0`)
- Déplacer `laravel/tinker` vers `require-dev`

### 12.2 Contrainte PHP

```json
"php": "^8.2"
```

PHP 8.2 a atteint sa fin de support actif en décembre 2024. La contrainte devrait être relevée à `^8.4` pour garantir l'utilisation de versions maintenues.

### 12.3 Race condition — Numéro de membre — MEDIUM

**Fichier :** `Member.php:22-25`

```php
static::updating(function (Member $member) {
    if ($member->isDirty('statuscode') && ... && !$member->member_number) {
        $maxNumber = (int) static::max('member_number');
        $member->member_number = str_pad((string) ($maxNumber + 1), 4, '0', STR_PAD_LEFT);
    }
});
```

L'attribution du numéro de membre n'est pas atomique. Deux requêtes concurrentes pourraient obtenir le même `max()` et attribuer un numéro en doublon.

**Recommandation :** Utiliser `DB::transaction()` avec `lockForUpdate()` ou une contrainte UNIQUE en base de données (déjà en place) combinée avec un retry en cas d'erreur.

## 14. Points forts de sécurité

L'application intègre plusieurs bonnes pratiques notables :

1. **Tokens cryptographiquement sûrs** — `bin2hex(random_bytes(32))` pour tous les tokens
2. **Hachage des tokens** — Seul le hash SHA-256 est stocké en base, jamais le token brut
3. **Tokens à usage unique** — Les magic links et tokens d'activation sont invalidés après usage
4. **Expiration des tokens** — 15 min (magic link), 72h (activation), 5 min (QR code)
